Phase 1

Introduction & Basics

Every time you write a program — whether it's in C, Java, or Python — you're writing something a human can read. But the computer only understands binary: zeros and ones. So how does your code actually become something the machine can run? The answer is a compiler.

Today, our team is going to build a mini compiler — from scratch — for arithmetic expressions. And we're going to trace one single expression through every stage of the compilation process.

Why this expression?

result = (a + b) * (c - d) / e

Because it has everything that makes a compiler interesting:

  • It has parentheses — which force grouping.
  • It has multiple operators — addition, subtraction, multiplication, and division — each with different precedence.
  • It has five variables and an assignment.

This means the compiler has real work to do at every stage. You'll see the same expression transform — first into 15 tokens, then into an abstract syntax tree, and finally into clean intermediate code.

Phase 2

Lexical Analysis (Scanner)

Thank you. I'm the Scanner — the very first phase of the compiler. My job is to read the source code character by character and split it into tokens.

Think of it like reading a sentence and identifying every word, punctuation mark, and symbol separately. You don't process 'The cat sat' as one lump — you see three distinct words. I do the same thing for code. A token is the smallest meaningful unit in the language — an identifier, a number, an operator, a parenthesis. All whitespace is discarded — spaces have no meaning here.

💡 How it matches tokens

The scanner uses a Finite Automata. It starts in an initial state, stringing characters together.

When it sees r, e, s, u, l, t, it stays in the "Identifier" state. Upon hitting a space or =, it reaches an accepting state, emits the token ID(result), and resets.

Phase 3

Parsing

I've received the token stream. Now my job is to answer two questions: Is this expression grammatically valid? And what is the correct structure?

Parsing is about understanding relationships. The tokens alone don't tell you whether * should happen before +, or which operands belong to which operator. The parser figures that out using a formal grammar — a set of rules about how expressions are structured.

Grammar Precedence Rules

Expression Joined by + or - (Lowest)
→
Term Joined by * or / (Medium)
→
Factor Variables, Numbers, or ( ) (Highest)

Because terms are nested inside expressions, * and / bind more tightly than + and -. And parentheses, being factors themselves, override everything — they force their contents to be evaluated first.

Phase 4

Intermediate Code Gen

Thank you. I have the AST, and my job is to flatten it into a sequence of simple instructions that a machine can eventually execute. We call this Three Address Code, or TAC.

Each instruction has the form: result = operand1 operator operand2. At most one operation per line. Temporary variables — t1, t2, etc. — are used to hold intermediate results.

Why not go straight to machine code? Because intermediate code is platform-independent. The same TAC can be compiled to x86, ARM, or any other architecture. It's also where optimizations happen in real compilers.
